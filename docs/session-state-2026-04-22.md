## 1. One-line status

A working UE5 packet emulator for Ashes of Creation that completes handshake
byte-perfectly, drives clients into the world via captured replay, and runs
a parallel generator pipeline (LiveWorld) that observes real traffic + produces
byte-identical NMT replies. The next bottleneck is synthesising the full
actor-spawn bunches (PC/Pawn/PlayerState) to replace replay-driven spawns.

## 6. Important RE findings

### UE5 NMT canonical codes (from `DataChannel.h` lines 173-194)
| Code | Name | Note |
|---|---|---|
| 0 | Hello | C→S initial |
| 1 | Welcome | S→C level+gamemode+redirect |
| 3 | Challenge | S→C challenge string |
| 5 | Login | C→S URL+online-id |
| **9** | **Join** | **NOT 7 — earlier drafts had 7 wrong** |
| **13** | **Abort** | **NOT 10** |
| **18** | **NetGUIDAssign** | **NOT 14. AoC reuses this for "game ready" signal (C→S)** |
| **20** | **GameSpecific** | stock UE5 value |

### AoC protocol deviations from stock UE5
1. **Skips NMT_Welcome and NMT_Join in the C→S flow** — client sends NMT_Hello → NMT_Login → NMT_Netspeed → NMT 18, jumping from AUTHENTICATED straight to IN_WORLD.
2. **Reuses opcode 18** as its "world ready" signal (UE5 defines 18 as NetGUIDAssign which is rarely used).
3. **Bunch-header `bControl=1`** only marks channel open/close bunches — regular NMT messages on the control channel have `bControl=0`. This is UE5-correct but my catalog classifier got it wrong initially; fixed.
4. **NMT uses 13-bit BunchDataBits and 10-bit ChSequence** (C→S-format framing) even in S→C direction on the control channel.
5. **Challenge string is a short decimal token** (e.g. `"50995344"`) rather than a 20-byte HMAC hex (the StatelessConnect HMAC is the cookie, not the challenge).

## 8. What's still blocking "play with our chosen character, no replay"

The user-visible gaps remain:
- ❌ Variable-length character name in HUD (replay path forces 10 chars)
- ❌ Class/race/gender selection takes effect (replay hardcodes RandomChar's Cleric)
- ❌ Correct HUD for chosen class (hotbar, skills, portrait come from replay)

All three unblock the same way: **Session G must own the actor-spawn bunches** (PC for HUD identity, Pawn with CharacterInformationComponent for nametag+class, PlayerState for HUD vitals) AND the subsequent property-delta stream. That requires H.3b through H.3f complete (byte-identity) + H.5b (replay skips those opcodes when Session G owns them).

Estimated work remaining to that milestone: 15-25 focused hours across 6-10 sessions.

## 9. Known issues / tech debt

- **`test_player_controller_builder` output label** says `[PASS] BYTE-IDENTICAL` but the test is actually a Phase 3.6 bit-count tolerance check, not true byte-identity. Cosmetic.
- **Catalog over-counts NMT_Hello** (2,733 false positives). Fix: skip bunches with `bdb<8` during classification. Deferred.
- **Outgoing fragmentation not implemented.** `UdpPacketEmitter::wrap_and_send` assumes one-bunch-per-packet. Needed for H.3e.
- **Legacy replay thread spawn** has a minor race at `replay_thread_.join()` followed by immediate re-assign. Low-risk, single-client only.
- **`ch_sequence_for_reliable` coordination** between legacy path + H.2 live-send needs finalization in H.5 (currently dry-run-only; see "counter-advance bug fix" from 2026-04-22).
