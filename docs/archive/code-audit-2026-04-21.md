## Finding 2 — MEDIUM: Triplicated `write_sip` (SerializeIntPacked)

**Evidence:** same 7-bit continuation encoding implemented in 3 places:

**Copy A** — `src/net/game_server.h:198-206`:
```cpp
inline void write_sip(uint8_t* buf, size_t cap, size_t& off, uint64_t value) {
    do {
        uint8_t byte = static_cast<uint8_t>((value & 0x7F) << 1);
        value >>= 7;
        if (value > 0) byte |= 1;
        write_bits(buf, cap, off, byte, 8);
    } while (value > 0);
}
```

**Copy B** — `src/net/ue5_replication.h:110-118` (as class method):
```cpp
void write_sip(uint32_t val) {
    do {
        uint8_t byte = static_cast<uint8_t>((val & 0x7F) << 1);
        val >>= 7;
        if (val > 0) byte |= 1;
        write_uint8(byte);
    } while (val > 0);
}
// ...and a separate write_sip64 at line 120 for uint64
```

**Copy C** — `src/protocol/bunch_builder.h:67-74`:
```cpp
void write_sip(uint64_t value) {
    do {
        uint8_t byte = static_cast<uint8_t>((value & 0x7F) << 1);
        value >>= 7;
        if (value > 0) byte |= 1;
        write_bits(byte, 8);
    } while (value > 0);
}
```

**Risk:**
- Any bug fix to the encoding (overflow handling, max-length enforcement) has to be applied in 3 places
- Changes to one and not others = subtle wire-format bugs that pass single-module tests

**Fix path:** consolidate into one canonical `ue5::` namespace function, call from the class methods. ~20 lines of edit across 3 files.

## Not-a-bug observations worth noting

### Bootstrap duplication explanation (my current hypothesis)

Reading the two usage sites more carefully:
- `src/data/bootstrap_data.h` is consumed by the **`Bootstrap loop`** that runs BEFORE the replay. It handles the early NMT phase with the older 500-packet dataset.
- `src/protocol/bootstrap/bootstrap_data.h` is consumed by **`BootstrapSequence::fill()`** which feeds `ReplayData` → `replay_loop` uses it.

So there are two distinct PHASES:
1. "Bootstrap" (pre-replay): sends the old 500 packets for NMT/handshake completion
2. "Replay" (post-NMT): sends the new 2000 packets for game-world loading

This is confusing but not incorrect — just poorly named. The old "bootstrap" is actually a **handshake completion kit**, and our new "bootstrap" is actually the **world-loading replay**.

Fix: rename `src/data/bootstrap_data.h` → `src/data/handshake_packets.h` and `bootstrap::BOOTSTRAP_PACKET_COUNT` → `handshake_packets::kPacketCount`. No behavior change, just clearer semantics.

### Why `patch_character_name` lives on `ReplayData`

Currently `ReplayData` owns the patch functions. This couples data to behavior. A cleaner separation would be a `ReplayPatcher` service. But pragmatically, the patches are tightly coupled to the specific packet layout of that ONE capture, so the coupling is defensible.
