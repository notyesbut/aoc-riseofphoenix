# Patchable Properties Catalog

Authoritative reference for what we can modify in AoC.
Last updated: 2026-04-26 (after full URepLayout RE — 12 functions decoded).

---

## Three patching tiers

We've identified **three distinct mechanisms** for changing what the
client sees, each with different scope and difficulty:

| Tier | Mechanism | What it controls | Difficulty |
|---|---|---|---|
| **Tier 1**  | Byte-pattern search/replace in replay bytes | FStrings (FString length-prefix is byte-aligned) | ★ Easy (working) |
| **Tier 1.5** | JSON metadata override (`data/characters.json`) + XClient proto | Character identity (name, class, race, level, gender) | ★★ TBD pending test |
| **Tier 2**  | Bit-level patcher using RepLayout cmd offsets | Replicated properties (HP, MP, stats) | ★★★★ Hard (2 days) |

**Insight discovered 2026-04-26**: our character's identity comes from
`characters.json` + UE5 client blueprints — NOT from replay bytes.
The replay only contains world state (other characters, NPCs, terrain).
Stats like HP=90 are likely **client-side blueprint defaults** based on
(class, level), not replicated values.

This means Tier-1.5 (JSON) may be sufficient for character-identity
changes, eliminating the need for Tier 2's bit-level work for many use
cases.

## Two data sources for character info — UPDATED 2026-04-26

After test #1 (level=5 worked, class swap didn't), we discovered the JSON
has **TWO sub-sources**: top-level fields vs create_info_hex blob.

### characters.json structure split

| Field type | Where | Updates from JSON top-level edit? | Notes |
|---|---|---|---|
| Display fields (level, gender_name, race_name) | top-level | ✅ YES | Verified test 1: level=5 lobby showed |
| Class identity (class_name, archetype_id) | **create_info_hex** | ❌ NO | Even though TOP-level changes, class display follows create_info_hex's embedded `"class":"<NAME>"` |
| Race identity (race_name) | **create_info_hex** | partial | Similar pattern |
| Customization (face, body, hair, decals) | **create_info_hex** | ❌ NO | Massive protobuf blob |
| Gender visual | **create_info_hex** | partial | |

### Verification (Hatemost test)

```
characters.json top-level:    archetype_id=17750, class_name="Mage"
create_info_hex internal:     "class":"Bard"
Lobby HUD:                    "Lvl 5 Bard" ← uses create_info_hex, ignores top-level
```

### Implication for class/race patching

To change class or race, we need to **decode + modify + re-encode the
create_info_hex protobuf**. Not impossible but more complex than a simple
JSON edit. Options:

1. **Use existing characters with desired class** (Hatemost=Bard,
   MyBigName=Mage) — no protobuf surgery needed.
2. **Add a new character entry** with the desired class, generated by
   the AoC client (then exported as create_info_hex).
3. **Build a protobuf editor** that decodes create_info_hex, modifies
   `"class":"X"` → `"class":"Y"`, re-encodes. Requires the proto schema.

### Bit-packing finding

Naive byte-pattern search across all 29,010 replay packets:

| Search | Hits |
|---|---|
| HP=90 as int32 (`5A 00 00 00`) | 0 |
| HP=90 as float (`00 00 B4 42`) | 0 |
| Stamina=100 as float (`00 00 C8 42`) | 0 |
| ClassId=17748 as int32 (`54 45 00 00`) | 0 |
| Adjacent HP+HP_max combo | 0 |

**Conclusion**: Properties of OTHER characters in the replay are
bit-packed (RepLayout writes at non-byte-aligned offsets). Tier-1
ReplayPropertyPatcher (byte search) cannot find them. **Tier 2 (bit
patcher) is required** for replayed-character or NPC stat patching.

## Bit-offset formula (for Tier 2 patcher design)

For property `P` at handle `N` in class `C`:

```
For each preceding property K (0..N-1):
  bits_consumed += SerializeInt_bits(NumProperties_in_C)   # cmd_handle field
  bits_consumed += SIP_bits(NumBits_for_K)                  # NumBits field
  bits_consumed += sum(cmd_bit_width(c) for c in K.cmds)    # value bits

# Property P's wire layout:
P_cmd_handle_offset = bits_consumed
P_NumBits_offset    = P_cmd_handle_offset + SerializeInt_bits(NumProperties)
P_data_offset       = P_NumBits_offset + SIP_bits(NumBits_for_P)
P_data_width        = sum(cmd_bit_width(c) for c in P.cmds)
```

To patch property P's value:
1. `write_bits(packet, P_data_offset, new_value, P_data_width)`
2. No bit shifts of subsequent bits (size unchanged for fixed-width)
3. No NumBits update needed (same bit count)
4. No bdb update needed (same total bunch size)

## What we can patch — three tiers ranked

### Tier 1 (works today — byte-aligned)

- ✅ `CharacterName` 1-10 chars (M2.1 NUL-pad replay byte-replace)
- ✅ Any other byte-aligned FString (FString[+0] is int32 length, then bytes)
- ✅ Fixed-position structure headers
- ❌ Any int32/float/bool RepLayout property (bit-packed, byte search fails)

### Tier 1.5 (test pending — JSON metadata)

- Hatemost: archetype 17747 (Bard), level 25
- MyBigNameCharact: archetype 17750 (Mage), level 25

If HUD shows Bard/Mage at level 25 after relaunch:
- ✅ `CharacterName` (already works via M2.1)
- ✅ `Level` (via JSON edit)
- ✅ `CharacterClass` / `CharacterClassId` (via JSON edit)
- ✅ `CharacterRace` / race_id (via JSON edit)
- ✅ Gender, customization (via JSON edit)

Then we extend the JSON schema + XClient proto handler to add:
- `Gold`, `Experience`, stats, etc. (server-side change, no RE needed)

### Tier 2 (2 days work — bit-level)

Once Tier 2 patcher is built (binary parser + bit-write):
- ✅ `Health`, `MaxHealth`, `Mana`, `MaxMana`, `Stamina`, `MaxStamina` (replayed chars)
- ✅ `Damage`, `BaseDamage`, `Resistance`, all combat properties
- ✅ `MoveSpeed`, `JumpHeight`, all movement
- ✅ Any replicated int32/float/bool property of any actor

### Tier 3 (long-term — FastArray emitter)

- Inventory contents (replicated as FastArray)
- Equipped gear
- Known abilities / hotbar
- Active buffs / debuffs (with durations)
- Quest log entries
- Spawn custom NPCs / items / props (NetGUID fabrication + PME)
