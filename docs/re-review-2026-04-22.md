## PART A — CustomDelta Re-verification

### What I scanned for (second pass)

All 305,099 printable strings in the 225MB exe, using regex patterns for:
- `FFast*Array`, `FFast*Serializer` — all FastArraySerializer variants
- `*::(PostReplicatedAdd|PostReplicatedChange|PreReplicatedRemove)` — all replication lifecycle hooks
- `*::NetSerialize`, `*::NetDeltaSerialize` — all class-specific overrides
- `OnRep_*` — all replicated property callbacks
- `DOREPLIFETIME` — UE5 replication-registration macros
- `C:\P4\*.cpp`, `C:\P4\*.h` — Perforce source-file paths

### Confirmed: FastArraySerializer use in AoC is VERY LIMITED

```
FFast*Array/Serializer found: 1 unique type
  FFastActorLocationArray  @ 0xb831e50
```

**Only `FFastActorLocationArray` exists.** Zero other FastArraySerializer types in the binary. No `FFastBuffArray`, no `FFastInventoryArray`, no `FFastAbilityArray`. They are not AoC's primary replication mechanism.

### Lifecycle hooks — only one type actually uses them

```
FFastActorLocationArray::PreReplicatedRemove     @ 0xb831ce0
FFastActorLocationArray::PostReplicatedAdd       @ 0xb831e50
FFastActorLocationArray::PostReplicatedChange    @ 0xb831f98
```

No other class exposes these hooks as strings. So **FastArraySerializer is used exclusively for actor-location tracking in AoC.** Everything else (abilities, inventory, stats, buffs) uses standard RepLayout or custom NetSerialize hooks we'd need to find differently.

### Confirmed: no per-class NetSerialize / NetDeltaSerialize overrides

```
*::NetSerialize overrides found in strings: 0
*::NetDeltaSerialize overrides found: 0
```

Virtual methods don't emit their own RTTI strings — they live as vtable slots. The only reason the 3 FastActorLocation hooks ARE visible as strings is debug-mode `check()` macros that reference `__FUNCTION__` — standard UE5 pattern. Nothing else triggers those check macros = no other class has custom NetDeltaSerialize logic leaking strings.

### IDB (CacheDB.dbc) format — partial reverse engineering

Header (first 48 bytes):
```
Offset  Bytes                                     Interpretation
0x00    49 44 42 01                              Magic "IDB" + version 1
0x04    74 c0 63 09                              unknown u32 = 0x0963c074
0x08    05 e0 6e 69                              unknown u32 = 0x696ee005 (looks like random seed)
0x0c    00 00 00 00                              padding
0x10    00 00 00 00 71 01 00 00                  u32=0 then u32=369 (possibly entry count?)
0x18    00 00 00 00 00 00 00 00                  zeros
0x20    00 00 00 00 00 00 00 00                  zeros
0x28    80 83 b0 07 00 00 00 00                  u32=0x07b08380 = 129,008,512 ← points to string table
0x30    (data rows start)
```

**Entries seem to be 16 bytes each starting at 0x30:**
```
Entry 0 @ 0x30: bc 0c e6 00 5b 58 c0 65 5c 2e 00 00 b7 2c 55 dd
Entry 1 @ 0x40: f1 b7 8c 19 5c 2e 00 00 00 00 00 00 d9 e0 1d 00
Entry 2 @ 0x50: 00 00 00 00 52 08 00 00 1f ed c4 7d 6f 51 c3 19
```

Could be interpreted many ways. Odd vs even rows show different patterns — possibly paired (index + metadata) blocks.

**At offset 0x07b08380 (from header field 0x28):**
```
3f 00 00 00 2f 47 61 6d 65 2f 55 49 5f 4b 69 74 ...
       |---|  / G  a  m  e  /  U  I  _  K  i  t ...
       = int32 length 0x3f = 63 bytes
       then "/Game/UI_Kit/Textures/AbilityIcons/ICON_BowCombo.ICON_BowCombo"
```

**Conclusion: 0x07b08380 is the START of a string table where each string is `[uint32 length][bytes]`.** The row table at 0x30 probably references into this string table.

Full IDB format decoding is an 8-12 hour project. For our MVP multiplayer purposes, grepping strings works fine.

### IoStore TOC — the real game content container

`global.utoc` (974 bytes, very small):
```
Magic: '-==--==--==--==-'   (UE5 IoStore TOC magic, verified)
Version: 0x08
Header size: 144 bytes
Entry count: 1                      ← just the global startup chunk
Compressed block size: 65536
```

Actual gameplay assets are in `pakchunk0-100+-WindowsClient.ucas/.utoc` pairs. These use UE5's standard IoStore format.

**Extraction tools (known working):**
- `retoc` — Rust CLI dedicated to .utoc/.ucas
- `FModel` — Windows GUI for UE5 assets
- `UnrealPak.exe` — official Epic tool (requires engine build)

Each extracts `.uasset` files containing structured UE5 objects — including `UDataTable` rows for races, archetypes, stats, items. **This is the canonical path to structured game data.**

## PART C — What we're still missing

### Known unknowns

1. **Exact bit offset of each RepLayout handle** per component. We have 276 property names but not their handle indices. Solvable via targeted walker run on initial-spawn bunches (Phase 1.1 of master plan).

2. **Exact encryption keys for paks.** UE5 IoStore paks CAN be encrypted. We haven't tested whether AoC's paks are. If encrypted, we'd need the AES key extracted from the client. Extraction tools handle this if provided the key.

3. **IDB row format.** We know entries are 16 bytes at 0x30 onwards. Full row-to-data mapping still to be done.

4. **Proto definitions of InterServer RPCs.** If we want to fake cross-server interactions (vassalship, caravans crossing server boundaries), we'd need the actual RPC proto schemas.

5. **The actual vtable offsets** for key virtual methods. Requires engaging IDA .i64 database (1.9GB, needs IDA Pro to open).
